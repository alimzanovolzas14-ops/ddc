# Бэкенд портала ЦЦР (реальный сервер)

Настоящий сервер для портала: **реальная аутентификация**, общий чат и личные
сообщения, заявки, и **мгновенная синхронизация между пользователями** через
WebSocket. Сервер также раздаёт сам сайт, поэтому всё работает с одного адреса.

Технологии: Node.js + Express + `ws` (WebSocket). Пароли хешируются через
встроенный `crypto` (scrypt).

**Два варианта хранилища — сервер выбирает автоматически:**

- **JSON-файл** (`data/db.json`) — по умолчанию, если не задан `DATABASE_URL`.
  Работает из коробки, без нативных модулей и внешних баз. Отлично для демо и
  небольшой команды.
- **PostgreSQL** — если задан `DATABASE_URL`. Это путь для продакшена: безопасная
  одновременная запись, индексы, транзакции, масштаб. Код один и тот же —
  меняется только место хранения.

## Запуск

```bash
cd backend
npm install
npm start
```

Откройте **http://localhost:3000** — это и есть сайт с порталом, уже подключённый
к серверу.

Демо-входы: `admin/admin`, `a.arinova/demo`, `m.amardinov/demo`,
`e.durmagambetov/demo`, `soc/soc`.

## Как включить бэкенд во фронте

Сервер сам раздаёт сайт из папки `../DDC_site`, и фронт определяет адрес
автоматически. Если вы открываете сайт отдельно (например, с другого адреса),
впишите адрес сервера в `DDC_site/config.js`:

```js
window.DDC_BACKEND = 'http://localhost:3000';
```

Если оставить пустым — портал работает в **локальном демо-режиме** (данные в
браузере), как и раньше. То есть бэкенд полностью опционален.

## Проверить, что realtime работает

Откройте портал в **двух** окнах (или на двух устройствах в одной сети),
войдите разными пользователями и напишите в общий чат — сообщение появится у
второго мгновенно, без перезагрузки.

## PostgreSQL (продакшен)

Достаточно задать `DATABASE_URL` — сервер сам создаст таблицы и наполнит их
начальными данными при первом старте.

```bash
# пример: локальный Postgres
createdb ddc
DATABASE_URL="postgres://localhost:5432/ddc" npm start
```

Схема (создаётся автоматически):

- `users(id, login UNIQUE, name, dept, role, pass_hash, last_seen)` — учётные записи;
- `items(seq, id UNIQUE, coll, from_login, to_login, data JSONB)` — по строке на
  каждую запись любой коллекции (chat, dm, requests, …); тело записи лежит в
  `data` (JSONB), а `from_login`/`to_login` вынесены для фильтрации личных
  сообщений.

Логика хранилища изолирована в `db-pg.js` (Postgres) и `db-json.js` (файл);
`db.js` просто выбирает нужный по наличию `DATABASE_URL`. Остальной сервер об
этом не знает.

## Переменные окружения

| Переменная     | По умолчанию            | Назначение                                       |
|----------------|-------------------------|--------------------------------------------------|
| `PORT`         | `3000`                  | Порт сервера                                     |
| `DATABASE_URL` | —                       | Строка подключения PostgreSQL. Пусто → JSON-файл |
| `DDC_SECRET`   | случайный при старте    | Секрет для подписи токенов (**задайте в проде**) |
| `SITE_DIR`     | `../DDC_site`           | Папка со статическим сайтом                       |

```bash
DDC_SECRET="длинная-случайная-строка" PORT=8080 npm start
```

Сбросить базу к начальным данным: `npm run seed`.

## API (кратко)

| Метод | Путь                 | Описание                                  |
|-------|----------------------|-------------------------------------------|
| POST  | `/api/login`         | `{login,password}` → `{token, user}`      |
| GET   | `/api/state`         | снимок состояния для текущего пользователя |
| POST  | `/api/:coll`         | добавить запись (chat, dm, reqs, …)       |
| PATCH | `/api/:coll/:id`     | обновить (реакции, «прочитано», статус)   |
| WS    | `/ws?token=…`        | поток событий realtime                    |

## Что нужно для продакшена

Это рабочая основа, а не «прод из коробки». Перед публикацией:

- поставить за HTTPS (например, за Nginx/Caddy) — WebSocket станет `wss://`;
- задать постоянный `DDC_SECRET`;
- включить PostgreSQL (задать DATABASE_URL) вместо JSON-файла — уже поддерживается;
- добавить ограничение частоты запросов и валидацию входных данных;
- при необходимости — httpOnly-cookie вместо токена в localStorage.
